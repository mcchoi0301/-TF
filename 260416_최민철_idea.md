# NAS 데이터 관리 시스템 (최민철이 Claude Cowork의 도움을 받아 작성)

-------------

## 0. 출발점: "어떤 질문에 답할 수 있어야 하는가"

아키텍처를 그리기 전에, 시스템이 **반드시 답할 수 있어야 하는 질문** 부터 정리한다. 
아키텍쳐의 모든 요소 - 디렉토리, 데이터 처리 파이프라인, 필수 기능 - 는 이 질문에 대한 답을 제공하는지 여부로 검증한다.

### 데이터 생산자의 입장
- "내가 지난 주 돌린 HPLC 결과, 지금 검색하면 내 이름과 날짜로 바로 찾아지나?"
- "내가 올린 파일이 제대로 분류됐나, 실패했으면 왜 실패했나?"

### 데이터 소비자의 입장
- "*Yarrowia lipolytica*로 **β-carotene** 생산한 실험의 HPLC raw data 있나?"
- "**lycopene** 정량에 쓴 표준곡선 데이터 어디 있지?"
- "ALE로 xylose 자화 향상시킨 데이터 — 세대별 OD 기록 있나?"
- "지난 3년간 발표한 논문 중 **CRISPRi** schematic이 들어간 figure 원본(.ai / .png / .jpeg) 다 보여줘."
- "작년 연차보고서에 쓴 '추진체계' 그림 재활용하고 싶은데, 원본 파일 어디?"
- "**비브리오섀시 과제** 2차년도 보고서에 인용된 raw data 리스트 만들어줘."
- "특허 `10-2022-0174175`의 figure와 그 figure가 나온 실험 raw data까지 찾아줄 수 있나?"

시스템이 이런 질문들에 답하려면 필요한 것은 자연스럽게 도출된다:
파일 타입 식별 / 통일된 어휘(균주명·화합물명·기술명) / 문서 ↔ figure ↔ raw data 연결 / 시간-사람-과제 축 기준 인덱스.

------------

## 1. 데이터 모델 — "파일"이 아니라 "엔티티와 레코드"

NAS의 모든 정보를 **엔티티(Entity)**, **레코드(Record)**, **관계(Relation)** 세 종류로 환원한다.
ISA 프레임워크(Investigation–Study–Assay, Nature의 데이터 전용 저널 Scientific Data에서 제시하는 포맷)의 개념을 차용하되 단순화한다.

### 엔티티 (Entity) — "이름이 붙는 것"
프로젝트 전체에서 **권한이 있는 단일 표기법**을 갖는 대상. 연구실 고유의 **authority file**에서 관리.
* Authority file : 도서관 등에서 제작 관리하는, canonical 이름과 alias(별칭)가 같이 적혀있는 파일
  ex.canonical:  Dostoevsky, Fyodor Mikhailovich (1821-1881)
     alias:
      - 도스토예프스키
      - 도스또예프스끼
      - Dostoyevsky, Fyodor
      - Fjodor Dostojevski
      - Достоевский, Фёдор Михайлович

| Entity       | Canonical name 예시                            | Authority 파일             |
|:-------------|:-----------------------------------------------|:---------------------------|
| Organism     | *E. coli* MG1655, *V. natriegens* ATCC14048    | `authorities/organisms.yaml` |
| Plasmid      | pET28a, pdCas9-mcr, pTrc99A                    | `authorities/plasmids.yaml`  |
| Compound     | lycopene, cadaverine, 3-HP, β-carotene         | `authorities/compounds.yaml` |
| Instrument   | Agilent 1260 HPLC, BD FACSAria III             | `authorities/instruments.yaml` |
| Person       | HSJ, SHB, MHW                                  | `authorities/people.yaml`    |
| Project      | 23_비브리오섀시, 26_인공유도진화                 | `authorities/projects.yaml`  |

> **왜 필요한가.** 실험자는 "V. natriegens", "Vnat", "나트리젠스"를 섞어 쓴다.
> authority 파일에 canonical 이름 + alias 목록을 두면, metadata 작성 시 알아서 canonical로 수렴된다.

### 레코드 (Record) — "데이터 생산자에 의해 시스템에 업로드되는 파일"
| Record Type | 설명                                    | 저장 위치 |
|:------------|:----------------------------------------|:----------|
| Assay       | 단일 측정 1건 (HPLC run 하나, FACS 1 tube 등) | `vault/assays/` |
| Asset       | 논문·특허·과제에 쓰인 산출물(figure, 본문, 첨부) | `vault/assets/` |
| Document    | 논문 원고, 특허 명세서, 보고서 등 문서 전체 | `vault/documents/` |

### 관계 (Relation) — "엔티티·레코드를 잇는 것"
`catalog/relations.jsonl`에 **한 줄당 한 관계**로 기록(append-only).

```
{"rel": "assay_uses_strain",    "assay": "ASY-20250812-HSJ-001", "strain": "ORG-Vnat-ATCC14048"}
{"rel": "figure_derived_from",  "asset": "FIG-2024-lycopene-F3", "assay":  "ASY-20230907-SHB-014"}
{"rel": "paper_cites_assay",    "doc":   "DOC-pub-046-lycopene", "assay":  "ASY-20230907-SHB-014"}
{"rel": "grant_includes_paper", "grant": "PRJ-21-우수신진",      "doc":    "DOC-pub-046-lycopene"}
```

이 한 파일만 있으면 "이 HPLC 데이터가 인용된 모든 논문·과제보고서" 같은 역방향 질문도 한 번에 풀린다.

## 2. 디렉토리 구조

세 영역으로 나눈다: **staging**(임시) / **vault**(영구·불변) / **catalog**(메타·인덱스).
코드는 별도 `engine/`에 두어 데이터 저장소와 섞지 않는다.

```
NAS_root/
│
├── staging/                              # [임시] 업로드한 파일이 임시로 머무르는 대기실. 검증, 분류, 승인이 끝나면 vault로 이동하고 비워짐.
│   └── {사용자ID}/
│       ├── assays/                       # 실험 raw data 업로드
│       │   └── {자유 폴더명}/
│       │       ├── {장비 export 파일들}
│       │       └── _draft.yaml           # 시스템이 자동 생성 → 사용자가 편집
│       └── assets/                       # 논문/특허/과제 관련 업로드
│           └── {자유 폴더명}/
│               ├── {docx/hwp/pdf/pptx/ai ...}
│               └── _draft.yaml
│
├── vault/                                # [영구] 검증을 통과한 원본 데이터의 영구 저장소. 한 번 들어오면 수정, 삭제 불가.
│   ├── assays/                           # 실험 레코드
│   │   ├── chromatography/
│   │   │   ├── GC/       {YYYY}/{MM}/    # ASY-YYYYMMDD-USER-NNN.*
│   │   │   └── HPLC/     {YYYY}/{MM}/
│   │   ├── spectroscopy/
│   │   │   ├── plate_reader/
│   │   │   ├── UV-Vis/
│   │   │   └── NanoDrop/
│   │   ├── sequencing/
│   │   │   ├── Sanger/                   # .ab1
│   │   │   ├── NGS_short/                # Illumina .fastq.gz
│   │   │   ├── NGS_long/                 # Nanopore/PacBio
│   │   │   └── plasmid_map/              # SnapGene .dna / GenBank .gb
│   │   ├── cytometry/FACS/               # .fcs
│   │   ├── imaging/
│   │   │   ├── gel/                      # agarose/SDS-PAGE
│   │   │   ├── microscopy/
│   │   │   └── colony_plate/
│   │   ├── fermentation/
│   │   │   ├── bioreactor/               # DO·pH·OD 시계열 로그
│   │   │   └── flask/                    # 엑셀 형태의 배양 기록
│   │   ├── qPCR/
│   │   ├── mass_spec/                    # LC-MS, MALDI
│   │   └── _quarantine/                  # 분류 실패 레코드
│   │
│   ├── documents/                        # 논문·특허·보고서 본문
│   │   ├── publications/{PaperID}/
│   │   ├── patents/{PatentID}/
│   │   └── grants/{ProjectID}/{차년도}/
│   │
│   └── assets/                           # 문서에 포함된 figure/표/본문 일러스트레이션
│       └── {AssetID}/
│           ├── source/                   # 편집 가능한 원본 (.pptx/.ai/.opj/.pzfx)
│           ├── export/                   # 논문 제출용 최종 (.tif/.pdf/.png)
│           └── panel_sources.json        # panel별 기반 raw data 목록
│
├── catalog/                              # [영구] 데이터 자체가 아닌, 데이터에 관한 정보(메타데이터, 인덱스)를 모아두는 도서관 목록실.
│   ├── authorities/                      # 통제 어휘(위 1절)
│   │   ├── organisms.yaml
│   │   ├── plasmids.yaml
│   │   ├── compounds.yaml
│   │   ├── instruments.yaml
│   │   ├── people.yaml
│   │   └── projects.yaml
│   ├── schemas/                          # 레코드 타입별 JSON Schema
│   │   ├── assay.hplc.schema.json
│   │   ├── assay.facs.schema.json
│   │   ├── asset.figure.schema.json
│   │   └── document.paper.schema.json
│   ├── records/                          # 레코드 개별 메타데이터
│   │   ├── assays/ASY-*.json
│   │   ├── assets/{FIG,TAB,TXT}-*.json
│   │   └── documents/DOC-*.json
│   ├── relations.jsonl                   # 모든 엔티티·레코드 관계 (append-only)
│   ├── views/                            # 자주 쓰는 질의를 precomputed index로
│   │   ├── by_compound.json
│   │   ├── by_organism.json
│   │   ├── by_project_year.json
│   │   └── figure_reuse.json
│   └── index/                            # 검색 엔진용 정/역인덱스
│       ├── tag_postings.json
│       ├── fulltext.sqlite               # 논문/특허 본문 전문 검색
│       └── caption.sqlite                # figure 캡션 검색
│
└── engine/                               # [코드] staging, vault, catalog 세 영역을 움직이는 처리 로직(파이프라인, 인식기, 검색엔진).
    ├── ingest/
    │   ├── identify.py                   # 파일 타입 식별
    │   ├── describe.py                   # 메타데이터 초안 작성
    │   ├── relate.py                     # 관계 그래프 업데이트
    │   └── publish.py                    # vault 이동 + 인덱스 갱신
    ├── recognizers/                      # 장비/문서별 인식기 (plugin)
    │   ├── hplc_agilent.py
    │   ├── facs_bd.py
    │   ├── ngs_fastq.py
    │   ├── figure_extract.py             # docx/hwp/pdf → 이미지 + 캡션
    │   └── ...
    ├── search/
    │   ├── tag_query.py
    │   ├── semantic_query.py
    │   └── lineage_query.py              # raw↔figure↔paper 추적
    └── ui/                               # CLI + 최소 웹 GUI
```

**설계상 포인트**

- **staging ≠ vault.** staging은 "아직 완전치 않다"는 시그널이 필요하므로 경로로 분리.
- **vault 아래에서만 ID로 부른다.** 파일 이름은 `ASY-20260416-HSJ-003.fcs` 형태로 **레코드 ID가 곧 이름**. 사용자 원본 이름은 메타에 보존만 한다.
- **catalog/relations.jsonl** 하나로 lineage 전부 추적. 그래프 DB를 처음부터 쓸 필요 없음(나중에 필요하면 덤프만 하면 된다).
- **authorities/** 는 한 번 정비해두면 "Vnat / V.nat / 나트리젠스" 같은 동의어 문제를 **근원적으로** 차단.

## 3. 처리 파이프라인 — 5-stage 변환

**각 단계가 순수 함수**인 파이프라인으로 설계한다.
어떤 단계든 독립적으로 재실행·테스트 가능해야 한다.

```
   [staging]
       │
  ┌────▼─────┐  1. IDENTIFY   파일·폴더가 무엇인지 판별
  │ identify │     (확장자 + 헤더 시그니처 + 폴더 구조)
  └────┬─────┘
       │
  ┌────▼─────┐  2. DESCRIBE   메타데이터 "초안" 작성
  │ describe │     (auto 필드 추출 + authorities 조회로 canonical화)
  └────┬─────┘
       │
  ┌────▼─────┐  3. CONFIRM    사용자 승인 or 편집
  │ confirm  │     (staging/{user}/.../ _draft.yaml ← 시스템이 채워둠)
  └────┬─────┘     (사용자가 열고 저장하면 통과 — 변경 없어도 OK)
       │
  ┌────▼─────┐  4. RELATE     엔티티·레코드 관계 등록
  │ relate   │     (relations.jsonl append)
  └────┬─────┘
       │
  ┌────▼─────┐  5. PUBLISH    vault로 이동 + catalog 인덱스 갱신
  │ publish  │     (이 단계 이후 파일은 read-only)
  └──────────┘
```

**3단계 CONFIRM이 존재하는 이유.**
실험자의 30초 수고(초안 열고 눈으로 확인 → 저장)가, 소비자의 10분 삽질(검색 실패 후 직접 뒤지기)을 막는다.
핵심은 "작성"이 아니라 "승인"이라는 점이다. 빈 템플릿을 채우게 하면 30명 중 절반이 대충 채우지만,
**채워진 초안을 검토**하는 건 싫어도 한다. 그리고 틀린 필드만 고치면 되니 실제 작성 분량이 0.5줄 수준이다.

### 3-A. IDENTIFY — 파일 정체 판별

```
file → [ext check] → [magic-bytes/header check] → [folder signature check]
                          │
                   3개 중 2개 일치 시 확정,
                   1개만 일치 시 confidence=low로 마킹
                   전부 불일치 시 _quarantine로
```

예시 시그니처:
- Agilent GC: `.D` 폴더 + 내부에 `FID1A.ch` / `REPORT.TXT`
- BD FACS: `.fcs` 파일 + header의 `$BEGINTEXT` 매직
- Illumina FASTQ: `.fastq.gz` + 첫 레코드의 `@` prefix + quality length 매치
- 논문 manuscript: `.word` + 본문에 "Abstract", "Results", "References" 키워드

### 3-B. DESCRIBE — 메타데이터 초안 자동 작성

인식기(recognizer)가 각 타입별로 필드를 채운다. **사람이 쓰기 싫어할수록 시스템이 더 채워야 한다.**

| Record type | auto가 채우는 필드 | human이 채울 필드 |
|:-----------|:------------------|:-----------------|
| HPLC       | 컬럼, 이동상, 파장, 유속, RT·peak area | strain, compound, medium, purpose |
| FACS       | 채널, gating, event count          | strain, sort criteria, downstream |
| NGS (FASTQ)| platform, read length, # reads, Q-score 분포 | library prep, sample biology |
| 현미경 이미지 | 해상도, 채널, scale bar 파싱      | sample, staining, time point     |
| 논문 원고   | 제목, 저자, 초록, 섹션 구조, 인용된 figure 목록 | 게재지, DOI(있으면), 관련 과제 |
| figure     | 포함된 panel 수, 캡션, figure type 추정 | source raw data IDs(복수 가능) |

### 3-C. figure 전용 세부 처리 — "provenance chain"

논문·보고서·특허 문서에 포함된 그림은 **단일 파일이 아니라 사슬**로 다룬다.

```
raw data (vault/assays/...)
   │  ─ 측정
   ▼
analysis tool output (Origin .opj / Prism .pzfx / Python notebook / Excel chart)
   │  ─ 시각화
   ▼
composited figure (PPTX 슬라이드 / Illustrator .ai)
   │  ─ 편집·레이아웃
   ▼
exported image (.tif / .png / .pdf)
   │  ─ 삽입
   ▼
document (.docx / .hwp / .pdf)
```

각 단계 산출물을 하나의 **Asset**으로 묶어 저장하고, `panel_sources.json`에 체인을 기록한다.

```json
{
  "asset_id": "FIG-046-lycopene-F3",
  "used_in":  ["DOC-pub-046-lycopene"],
  "figure_number": "Fig. 3",
  "caption": "HPLC quantification of lycopene produced by engineered strains.",
  "panels": [
    {
      "panel": "A",
      "figure_type": "bar_graph",
      "source_assays": ["ASY-20230907-SHB-014", "ASY-20230907-SHB-015"],
      "plot_source":   "source/Fig3_HPLC.pzfx",
      "composite":     "source/Fig3_assembled.pptx",
      "export":        "export/Fig3.tif"
    },
    { "panel": "B", "figure_type": "schematic",
      "composite": "source/pathway.ai", "export": "export/Fig3.tif" }
  ]
}
```

이 체인이 구축되면 "Fig. 3B 원본 일러스트 파일 줘" / "이 HPLC 데이터가 논문 어디 실렸지?" 같은 질문이 양방향으로 한 번에 풀린다.

## 4. 실험 데이터 측 상세

### 4-A. `_draft.yaml` — 사용자에게 건네는 30초짜리 확인 양식

`staging/HSJ/assays/20260416_run2/` 에 사용자가 장비 export 폴더를 통째로 떨어뜨리면, IDENTIFY·DESCRIBE를 지난 뒤 아래가 **자동 생성**된다.

```yaml
# 시스템이 채운 초안 — 맞으면 그대로 저장, 틀리면 고친 뒤 저장
record_id:   ASY-20260416-HSJ-003    # 확정 전까지 임시 ID
type:        chromatography/HPLC     # 확신: 높음 (확장자+헤더+폴더 모두 일치)
instrument:  Agilent 1260 HPLC       # authorities에서 canonical화됨
detected:
```

## 5. 필수 기능 제안

### 필수 기능 ①  Authorities (어휘 통일)
**없으면 나머지가 전부 흔들린다.** 균주명·화합물명·기술명·인명·과제명 등 authority 파일부터 손으로 정리하고, 모든 파이프라인이 여기서만 canonical 이름을 가져오도록 강제.
→ alias 파일 같이 관리(`V.nat`, `Vnat`, `나트리젠스` → `V. natriegens`).

### 필수 기능 ②  Recognizer plugin 구조
장비별·문서타입별 인식기를 **각자 한 파일**로 떨어뜨리는 plugin 구조. 새 장비가 들어오면 `engine/recognizers/`에 파일 하나만 추가하면 된다. 각 plugin의 공통 인터페이스:

```python
class Recognizer(Protocol):
    def matches(self, path) -> Confidence: ...
    def describe(self, path) -> dict: ...
    def schema_id(self) -> str: ...
```

### 필수 기능 ③  Draft-and-confirm UX
사용자 편집 가능한 `_draft.yaml`을 시스템이 먼저 채워서 건네는 방식.
핵심 원칙: **사람에게 백지를 주지 않는다.**

### 필수 기능 ④  relations.jsonl + lineage query
raw↔figure↔paper↔grant 전방향 추적. append-only 한 파일이라 구현 단순, 백업·복구 쉬움.

### 필수 기능 ⑤  Figure provenance 체인
`source/` + `export/` + `panel_sources.json`. 재사용 감지(pixel hash)는 처음엔 optional로 시작해도 된다.

### 필수 기능 ⑥  View 파일 (precomputed 인덱스)
`by_compound`, `by_organism`, `by_project_year`, `figure_reuse` 같은 뷰를 미리 만들어두면 CLI/GUI 응답이 즉시 나간다. 매일 밤 한 번 재생성.

### 필수 기능 ⑦  2단계 검색
- **L1: tag/authority exact match** — 90% 질의는 여기서 해결
- **L2: semantic fallback** — L1 결과 부족 시 LLM이 질의를 authority 축으로 분해 후 재검색

### 필수 기능 ⑧  알림 — 실패만, 하루 1통 요약
- 분류 실패 / 초안 미확정이 **본인 것**인 경우에만
- 실시간 알림 X, 오전 9시 요약 1통
- 과제 담당자에게만 월간 `figure_reuse` 리포트(논문·특허 재활용 현황) 발송

## 6. 우선순위

| 순서 | 항목                                        | 근거 |
|:---:|:--------------------------------------------|:----|
| 1 | **Authorities 파일 초기 정비**              | 모든 검색·분류의 기반. 데이터 쌓기 전에 먼저. |
| 2 | **레코드 스키마 + `_draft.yaml` 작성 UX**   | 데이터가 들어오기 시작하는 지점 |
| 3 | **HPLC·FACS·NGS recognizer 3종**            | 이 3개만 있어도 실험 데이터의 대다수 커버 |
| 4 | **vault/catalog 저장 + relations.jsonl**    | 관계 축적이 시작되면 검색이 곧바로 유용해짐 |
| 5 | **tag 기반 검색 (L1)**                       | 이 시점에 실사용 가능 — 팀에 공개 |
| 6 | **문서/figure 인식기 + provenance 체인**    | 논문·특허·과제 실적과 raw data 연결 |
| 7 | **precomputed views + 월간 재사용 리포트**   | 연차보고서·특허 작성 시 가치 폭발 |
| 8 | **semantic 검색 (L2)**                       | 데이터·authority가 충분히 쌓인 뒤 |

## 7. 핵심 설계 원칙

| 원칙                          | 한 줄 설명 |
|:------------------------------|:----------|
| Query-first                   | "어떤 질의에 답할지"를 먼저 고정하고, 거기서 거꾸로 설계 |
| Authority over convention     | 관행 말고 authority file이 유일한 진실 |
| Draft, don't blank            | 사용자에게 빈 양식 강제 금지 — 초안 채워서 승인만 요청 |
| RAW is immutable              | vault 진입 후 수정·삭제 불가. 모든 파생은 catalog/asset에서 |
| One file, one relation        | 관계는 jsonl 한 줄씩. DB 안 써도 처음부터 동작 |
| Plugins at the edge           | 장비·문서 타입 변동은 recognizer plugin 추가로 흡수 |
